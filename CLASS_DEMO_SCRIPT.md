# Agent Harness A/B 课堂演示讲稿

## 一句话开场

今天我演示的不是一个单纯的 Python 小项目，而是一个用于教学的 A/B 实验框架。  
它的核心目的，是比较在相同模型、相同提示词、相同时间限制下，普通代码仓库和带有 agent harness 规范文件的仓库，最终交付质量会不会出现明显差异。

---

## 适合边演示边讲的完整讲稿

### 1. 项目背景

大家现在看到的是整个实验工作区。  
这个项目叫 `Agent Harness A/B Demo`，它不是让学生单纯去修一个 bug，而是用一个真实但规模可控的任务，观察大模型在不同上下文条件下的表现差异。

这里的实验设计非常简单：  
我们准备两份几乎一样的代码仓库，一份给 A 组，一份给 B 组。  
两组面对的是同一个业务问题、同一个起始代码、同一个提示词、同一个模型、同一个时间限制。  
唯一的区别在于，B 组的仓库里额外放了一组规范文件，也就是我们这里说的 agent harness。

所以这个实验想回答的问题不是“模型能不能写代码”，而是“当模型拿到结构化规范之后，能不能做出更完整、更符合工程约束的修复”。

---

### 2. 目录结构介绍

现在我先给大家看一下目录结构。  
这个工作区主要分成四部分。

第一部分是 `baseline_repo/`。  
这个仓库给 A 组学生使用，里面只有任务说明、源码和 visible tests，也就是学生能直接看到的测试。

第二部分是 `harness_repo/`。  
这个仓库给 B 组学生使用。它和 baseline 的业务代码是一样的，但是额外增加了四类信息：

- `AGENTS.md`
- `ARCHITECTURE.md`
- `QUALITY.md`
- `docs/product-specs/new-user-onboarding.md`

这四个文件分别告诉模型：

- 先看哪些规范
- 代码应该放在哪一层
- 什么才算完成
- 真实业务规则到底是什么

第三部分是 `instructor_assets/`。  
这一部分是教师专用评测资源，学生看不到。里面有 hidden tests、架构检查脚本和评分脚本。

第四部分是根目录下的说明文件，比如 `TEACHER_GUIDE.md` 和 `EXPERIMENT_FLOW.md`。  
这些文件主要帮助老师控制实验流程，保证 A/B 对比尽量公平。

---

### 3. 任务本身是什么

接下来我说一下这个任务到底在修什么。  
表面上看，这是一个“新用户注册”的小功能，问题集中在用户名规范化，也就是 username canonicalization。

更准确地说，starter code 里当前有两个明确 bug：

- 第一，规范化逻辑没有做完整。现有实现基本只做了 `strip + lower`，但没有把连续空格或连字符折叠成单个下划线，也没有去掉规范化后首尾多余下划线。
- 第二，重复用户名判断发生得太早。系统现在是先拿原始输入去查重，再做规范化，所以像 `team_lead` 和 ` Team---Lead ` 这种本应视为同一个用户名的输入，可能会被错误地当成两个不同用户。

所以课堂上可以把这个 bug 讲得更具体一点：  
不是“字符串处理写得不太对”，而是“注册流程对 canonical username 的定义和使用不一致，导致规范化结果、保留名判断、长度判断和重复判断都可能出错”。

项目要求用户名在注册时做统一规范化处理，比如：

- 去掉首尾空格
- 转成小写
- 把连续的空格或者连字符折叠成一个下划线
- 再去掉首尾多余的下划线

而且规范化之后的用户名还要继续满足业务约束，比如：

- 长度必须在 3 到 20 之间
- 只能包含小写字母、数字和下划线
- `admin`、`root`、`system` 是保留名
- `admin_`、`root_`、`system_` 这类保留前缀也不能注册
- 重复用户名判断必须基于规范化后的 canonical username

也就是说，这个任务不是简单地把一个字符串改一改，而是把“规范化、校验、重复判断、分层放置、测试补充”一起做好。

---

### 4. 为什么这个任务适合做 A/B 实验

这个任务很适合做课堂实验，因为它有一个很典型的特点：  
visible tests 只覆盖了最表面的情况。

比如一开始学生能看到的测试，只能看出像 `Alice-Smith` 这种输入要变成 `alice_smith`。  
如果模型只盯着 visible tests，它很可能只会做一个最小修补，让当前测试通过就停下。

但真实业务规则其实更完整。  
比如连续分隔符折叠、保留前缀、规范化之后的重复用户名、规范化之后再做长度判断，这些如果没有额外规范引导，模型不一定会主动补全。

所以这个实验的关键不在于“会不会修 bug”，而在于“会不会修得完整、修得对、修得符合架构”。

---

### 5. baseline 组怎么做

现在假设我是 A 组学生，我只拿到 `baseline_repo/`。  
我进入仓库以后，先安装依赖，然后运行 visible checks。

这时候我会看到当前代码没有完全满足要求。  
接着我给模型的提示词非常简单，只有一句话：

`请修复注册流程中的用户名规范化问题，并补上必要测试。完成后确保现有测试通过。`

注意，这个提示词对两组是完全一样的。  
A 组没有额外规范文件可读，所以模型更多只能根据当前代码和 visible tests 来推断需求。

这就很容易出现几种情况：

- 只修 visible bug
- 测试补得不够
- 把规则写到了 service 层或者 api 层
- 为了过眼前测试，做了局部修补，但 hidden tests 不一定能过

从教学角度看，这一组代表的是“只有任务，没有结构化工程上下文”的情况。

---

### 6. harness 组怎么做

接下来我们看 B 组，也就是 `harness_repo/`。  
这组拿到的起始代码和 baseline 是一样的，提示词也完全一样。

但不同的是，模型在动手之前，可以先读仓库里的规范文件。  
比如 `AGENTS.md` 会明确告诉它：

- 先看规范再改代码
- 不要只修 visible failure
- 领域规则不要写进 API 层
- 最终说明里要讲清楚改了什么、为什么这么改、怎么验证

`ARCHITECTURE.md` 会继续约束代码位置，明确说用户名规范化只能放在 domain 层，service 只做流程编排，API 只做输入输出。

`QUALITY.md` 会定义完成标准，比如至少新增 3 个 visible tests，总测试数至少到 5，还要覆盖保留前缀、规范化后重复用户名、异常路径断言。

而产品规格文档会把真实业务规则说清楚。  
模型不需要靠猜，它可以直接依据文档实施修复。

所以 harness 组的优势不在于“更聪明”，而在于“上下文更完整，约束更明确，目标更可验证”。

---

### 7. 教师怎么评测

学生提交之后，教师不会只看 visible tests。  
教师会在工作区根目录执行统一评测命令，对学生提交做四类检查：

- visible tests
- hidden tests
- architecture check
- visible test expansion check

也就是说，最后比的不只是“跑通没跑通”，还包括：

- 有没有真正理解业务规则
- 有没有把规则放在正确分层
- 有没有主动补足测试
- 有没有为了过测试写出很脆弱的修补代码

这个设计很适合课堂展示，因为它能把“结果正确”和“工程质量正确”区分开来。

---

### 8. 课堂上我会重点强调什么

如果我是老师，在演示过程中我会重点强调下面三点。

第一，A/B 两组的唯一差异必须严格控制。  
同一个模型、同一个提示词、同一个时间限制、同一个起始代码，只有 harness 文件不同。  
只有这样，最后的差异才有解释力。

第二，这个项目想证明的不是 harness 一定万能，而是规范文件会显著影响模型的决策路径。  
模型拿不到业务规则时，更容易做局部修补；  
模型拿到结构化规则时，更容易做完整修复。

第三，这个实验本质上是在讨论“如何给模型提供工程上下文”。  
我们不是只问模型会不会写代码，而是在问：  
如果我们把任务目标、架构边界、质量标准、产品规则都写清楚，模型是不是能更稳定地交付符合要求的结果。

---

### 9. 结尾总结

最后我会这样收尾：

这个项目虽然规模不大，但它把一个很有代表性的工程问题压缩进了课堂环境里。  
我们通过一个用户名注册的小任务，去观察大模型在不同上下文供给条件下的行为差异。

`baseline_repo` 代表“只有代码和任务”。  
`harness_repo` 代表“代码、任务，再加上明确的工程规范和产品规则”。

最终我们希望学生看到的是：  
影响模型输出质量的，不只是模型本身，还有我们给它提供的上下文结构。  
而 agent harness 的价值，就体现在把这些原本零散、隐性的经验，变成模型可读取、可执行、可验证的工作约束。

## 5 分钟精简版口播

大家现在看到的是一个用于课堂教学的 A/B 实验项目。  
这个实验想比较的是，在相同模型、相同提示词、相同时间限制下，普通仓库和带有 agent harness 规范文件的仓库，会不会产生不同的修复结果。

这个工作区里有两个学生仓库。  
`baseline_repo` 给 A 组，只有源码、任务和 visible tests；  
`harness_repo` 给 B 组，业务代码相同，但多了 `AGENTS.md`、`ARCHITECTURE.md`、`QUALITY.md` 和产品规格文档。

两组面对的任务都是修复注册流程中的用户名规范化问题。  
表面上看只是一个字符串处理 bug，但实际涉及规范化规则、保留前缀、重复用户名判断、测试补充以及代码分层。

如果要在演示时把 bug 说得再落地一点，可以直接说：  
当前 starter code 只做了最粗糙的 `trim + lowercase`，没有完整生成 canonical username；  
同时它在规范化之前就查重，所以同一个用户名的不同写法有机会绕过重复校验。

这个任务的关键在于，visible tests 只覆盖了一部分情况。  
如果模型只盯着眼前测试，往往只能做局部修复；  
如果模型能读到架构约束、质量标准和业务规格，就更有机会做出完整修复。

最后教师会用统一脚本评测四个维度：visible tests、hidden tests、架构检查和测试扩展。  
所以这个项目真正想展示的是，影响模型交付质量的，不只是模型能力本身，还包括我们是否给它提供了清晰、结构化、可执行的工程上下文。

---

## 演示时建议的操作顺序

1. 先在根目录介绍整体结构。
2. 打开 `README.md`，说明这是 A/B 教学实验。
3. 展示 `baseline_repo/` 与 `harness_repo/` 的目录差异。
4. 打开 `harness_repo/docs/product-specs/new-user-onboarding.md`，讲清业务规则。
5. 打开 `harness_repo/ARCHITECTURE.md` 和 `QUALITY.md`，讲清分层和完成标准。
6. 展示 prompt：`请修复注册流程中的用户名规范化问题，并补上必要测试。完成后确保现有测试通过。`
7. 说明教师最终会用 `instructor_assets/evaluate_submission.py` 做统一评测。
8. 用结尾总结把“模型能力”和“上下文设计”联系起来。
